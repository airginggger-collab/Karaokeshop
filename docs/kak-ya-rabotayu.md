# Как я работаю — скрипт запуска проекта

> Методология на основе реального опыта разработки karaokeshop.kz.  
> 6 этапов от первого звонка до передачи заказчику.

---

## Этап 1 — Интервью с заказчиком (30–60 мин)

**Цель:** понять бизнес, аудиторию и ограничения ДО того как открыт редактор.

### Вопросы:

**Бизнес и цель**
- Расскажи про бизнес в 3 предложениях. Кто покупает — частники или компании?
- Что сейчас не работает на старом сайте (или без сайта)?

**Главная цель сайта** *(выбрать одну)*
- Если сайт сделает одно — что это должно быть?
- Как ты поймёшь что он работает? (звонки / заявки / позиции в поиске)

**Ограничения**
- Бюджет хостинга — платить ежемесячно или нет?
- Дедлайн — жёсткий или желаемый?
- Что точно НЕ нужно? (авторизация, корзина, CMS, рассрочка…)

> ⚠️ **Ловушка:** Kaspi-рассрочку в karaokeshop добавили без спроса — потом удаляли из 21 места.  
> Спроси про запреты ДО начала.

## Этап 2 — Брейншторм и стратегия (с Claude, 1–2 часа)

**Итог:** файлы в `docs/strategy/`

### Шаги:

**Анализ конкурентов**
```
«Проанализируй конкурентов [ниша] в [город/страна].
 Что они делают плохо? Где наше окно?»
```

**URL-карта от семантики**
```
«Составь URL-карту сайта от семантического ядра для [ниша + гео].
 SEO-приоритет. Каждый URL = кластер поисковых запросов.»
```

**Архитектурное решение**
```
«Предложи архитектуру с учётом: бюджет хостинга — минимальный,
 SEO — главное, CMS — потом.»
```

Зафиксировать решения в ADR (Architecture Decision Record) — чтобы не переизобретать через месяц.

## Этап 3 — Утверждение визуала (до написания кода)

**Правило:** заказчик пишет «ок» на конкретный мокап. Скриншот → `docs/design/`.

### Шаги:

**Дизайн-токены и палитра**
```
«Предложи 3 варианта палитры для [ниша].
 Покажи как они выглядят на кнопке, карточке, хедере.»
```

**Мокап главной (виджет в чате)**
```
«Нарисуй мокап главной страницы: hero, 3 секции ниже, футер.
 Стиль: [референс или описание].»
```

**Письменное утверждение**  
Скриншот + «ок от [имя] [дата]» → защита от «я не это имел в виду».

> ⚠️ **Ловушка:** В karaokeshop дизайн менялся 3 раза (clay morphism → editorial → bento)  
> потому что не было утверждённого мокапа на старте.

## Этап 4 — Сборка итерациями

**Правило коммита:**
```bash
npm run build && npm test   # сначала
git commit                  # только если зелёное
git push                    # сразу после коммита
```

### Три итерации:

| Итерация | Фокус | Результат |
|---|---|---|
| 1 — Скелет | Монорепо, роуты, CI/CD | Живой URL на день 1 |
| 2 — Контент | Все страницы, SEO-метаданные, JSON-LD | Google видит сайт |
| 3 — Дизайн | Полировка UI, тёмная тема, анимации | Выглядит как задумано |

**Каждый коммит обновляет `docs/HANDOFF.md`** — следующая сессия стартует с контекстом.

---

## Этап 5 — Проверка перед сдачей

### Чеклист:

- [ ] **SEO:** Lighthouse ≥ 0.95. Все страницы в sitemap. JSON-LD валидный
- [ ] **Мобильный:** скриншоты 375px и 1280px, обе темы
- [ ] **Контент-аудит:** `grep -r "TODO\|FIXME\|XXXXXXXX\|placeholder" src/`
- [ ] **Аналитика:** Umami / Яндекс.Метрика с реальным ID, события фиксируются
- [ ] **CTA:** все кнопки ведут куда надо (WhatsApp, форма, телефон)

## Этап 6 — Передача заказчику

| Что | Зачем |
|---|---|
| Инструкция (PDF / .docx) | Менять текст и фото без разработчика |
| Доступы (GitHub, хостинг, домен) | Заказчик владеет своим сайтом |
| Backlog незакрытых задач | Без сюрпризов после сдачи |
| Кейс для портфолио | Пока свежо — метрики, скриншоты до/после |

---

## Почему это работает

Этот процесс — не теория. Он собран из реальных ошибок:

- Добавили фичу без спроса → пришлось удалять из 21 места
- Не утвердили дизайн → 3 редизайна
- Пушили без сборки → CI красный, заказчик нервничает
- Не обновляли HANDOFF → следующая сессия начиналась с нуля

**Правило одно: сначала договорились — потом сделали.**
