# Karaokeshop.kz — кейс для портфолио

> Полный цикл разработки интернет-магазина с нуля: от идеи до продакшена за одну сессию.  
> Автор: первый проект, написанный совместно с Claude, без предыдущего опыта веб-разработки.

---

## Коротко

**Что:** полный ребилд сайта интернет-магазина профессионального карaоке-оборудования. Алматы. Официальный дилер AST (Art System) и Studio Evolution.

**Стек:** Next.js 15 · TypeScript · Tailwind CSS · Turborepo · Cloudflare · GitHub Actions CI/CD

**Результат:** живой сайт на [karaokeshop.kz](https://www.karaokeshop.kz) — 77 статических страниц, автодеплой на каждый пуш, полноценный SEO-слой, калькулятор сметы, каталог 18 товаров, 26 статей блога.

**Особенность:** проект написан с нулевыми знаниями фронтенда. Каждый технический термин, каждое архитектурное решение — было новым. Это история о том, как задавать правильные вопросы.

## Что построили

### Архитектура

Выбрали **монорепо Turborepo + npm workspaces** — не потому что «так надо», а потому что сразу видели рост: отдельный дизайн-кит, отдельные токены цветов, отдельный сайт. Каждый пакет живёт сам:

```
apps/web/          → сайт (Next.js 15 App Router)
packages/tokens/   → @kk/tokens: дизайн-токены (цвета + радиусы)
packages/ui/       → @kk/ui: переиспользуемые компоненты (Button, Badge)
```

**Почему статика (`output: export`)?** Заказчик не хочет платить за сервер. Cloudflare Workers — бесплатно, молниеносно, надёжно. Переезд на SSR (если понадобится CMS) — это смена одного флага, не рефакторинг.

### Дизайн-система без дизайнера

Всё на **дизайн-токенах**: цвета объявлены один раз в `packages/tokens/css/theme.css`, оттуда импортируются и в сайт, и в Storybook. Два слоя акцента:
- `--warm-*` — тёплая палитра для домашнего сегмента
- `--night-*` — деловая тёмная для B2B / заведений
- `--color-*` — нейтральный базовый слой, флипается в `.dark`

Tailwind сидит поверх токенов. `text-foreground` вместо `text-[#333]` — тема меняется автоматически.

### SEO — позвоночник проекта

С самого начала SEO был главным приоритетом — заказчик хочет органику, не рекламу.

- **JSON-LD:** LocalBusiness (глобально), Product + AggregateRating на каждом товаре, BreadcrumbList, FAQPage на 5 страницах
- **Семантическое ядро:** URL-карта строилась от кластеров запросов («карaоке для дома Алматы», «карaоке для ресторана», «AST-250 купить»)
- **Блог:** 26 статей по ~5 000 символов каждая — гайды, сравнения, гео («карaоке в Астане»), how-to
- **Хлебные крошки** на каждой странице (конвенция закреплена в CLAUDE.md)
- **Авто-новости:** build-time fetch Google News RSS → бакается в статику. Ежедневный rebuild через GitHub Actions cron

### CI/CD без DevOps-опыта

Автоматизировали с нуля:

```yaml
push → main
  → npm ci
  → токены + тесты + сборка + Storybook
  → npx wrangler deploy   # автодеплой в Cloudflare
```

Параллельно — Lighthouse CI (6 URL). **Сюрприз:** performance — `warn`, не `error`. Иначе каждый пуш выглядел как «упал CI», хотя деплой прошёл.

Cron `0 8 * * *` — ежедневный rebuild для обновления новостей.

## Путь: ошибки и то, чему они научили

### Ошибка 1 — два источника правды для цветов

Цвета объявлялись в `tokens.json` и в `globals.css` одновременно. `globals.css` тихо перебивал токены. Симптом: «правлю цвет в одном месте — ничего не меняется».

**Вывод:** один источник на один вид токена. Цвета — только `theme.css`. Это правило теперь записано в CLAUDE.md.

### Ошибка 2 — `darkMode: "class"` не был включён в Tailwind

Dark-утилиты (`dark:bg-...`) работали через media query вместо `.dark` класса. Тёмная тема по кнопке не работала — только по системной теме.

**Вывод:** читать документацию до конца, особенно про конфигурацию.

### Ошибка 3 — `<Mood variant="night">` форсировал цвет независимо от темы

Несколько страниц были обёрнуты в компонент `Mood`, который ставил тёмную палитру принудительно. В светлой теме — тёмный «остров» внутри светлого шелла. Выглядело сломано.

**Вывод:** если компонент меняет тему, он должен следовать теме, а не подменять её. Компонент `Mood` удалён полностью.

### Ошибка 4 — WhatsApp-кнопка прямо на карточке товара

На старте добавили большую зелёную кнопку «Узнать цену» прямо на карточку. Выглядело навязчиво, дублировало возможность на странице товара, мешало клику по карточке.

**Вывод:** конверсионные элементы — на странице товара, карточка — только для навигации.

### Ошибка 5 — иконки в карточках роняли рендер при удалении

Тип `icon: LucideIcon` (не опциональный) → удаление иконки из данных = `undefined` → краш рендера → ошибка сборки.

**Вывод:** поле `icon?: LucideIcon` (опциональное) + guard `{Icon && <Icon />}` в рендере. Теперь иконку можно убрать из данных не трогая компонент.

### Ошибка 6 — коммит в чужой репозиторий

Bash сбрасывает рабочую директорию между вызовами. Один раз коммит ушёл в репозиторий соседнего проекта (`medlog`).

**Вывод:** все git-команды — через `git -C ~/Desktop/karaokeshop`. Закреплено в CLAUDE.md как первая ловушка.

### Ошибка 7 — Kaspi-рассрочка: ~21 вхождение после решения об удалении

Решение «убрать рассрочку» было принято, но код и SEO-описания разошлись — рассрочка осталась в hero, карточках, FAQ, блоге, метаданных.

**Вывод:** контент и код — единый источник правды. После решения нужен grep + полный аудит, не точечная правка.

## Что придумали (приятные открытия)

**Авто-новости без CMS.** Статика не мешает свежести. Build-time RSS fetch + ежедневный cron = новости обновляются сами, без сервера и без базы данных.

**Дизайн-токены как защита от будущих ошибок.** Один файл `theme.css` — все цвета сайта. Смена бренда = правка в одном месте.

**Калькулятор сметы.** 3-шаговый wizard рассчитывает конфигурацию под площадь и бюджет, отправляет итог в WhatsApp одним нажатием. Заказчик закрывает сделку быстрее.

**«Истории установок».** Masonry-лента в стиле соцсетей на странице блога: аватар-инициалы, хештеги, лайки. Социальное доказательство без реальных аккаунтов — показывает реальные кейсы.

**Lottie-анимация.** Добавляет живости герою, весит 70KB, не блокирует рендер (клиентский компонент, загружается после hydration).

**SearchOverlay.** Поиск в хедере фильтрует товары и статьи блога прямо в оверлее, без перехода на отдельную страницу.

## Техника, которую освоили по ходу

| Концепция | Как столкнулись |
|---|---|
| App Router vs Pages Router | Выбор на старте — App Router, потому что будущее |
| RSC (React Server Components) | ProductCard стал серверным после удаления `onClick` |
| `output: export` и его ограничения | Нет ISR, нет API routes, нет `next start` |
| Дизайн-токены + Style Dictionary | Один источник правды для цветов и радиусов |
| JSON-LD / Schema.org | LocalBusiness, Product, FAQ, Breadcrumb — для Google |
| Cloudflare Workers vs Pages | Разница между статическим Worker и Connect-to-Git |
| GitHub Actions | CI/CD с нуля: build → test → deploy |
| WCAG AA контраст | Считали соотношение, правили токены до ≥4.5:1 |
| Turborepo | Монорепо, workspace-команды, граф зависимостей |
| `next/font/google` | Manrope + Unbounded без layout shift |

## Чему этот проект учит про AI-разработку

Это не «ИИ написал сайт». Это «человек без опыта построил сложную систему, потому что умел объяснить задачу и оценить результат».

Ключевые навыки, которые оказались важнее знания синтаксиса:
- Декомпозиция: разбить «сделай сайт» на 39 конкретных задач
- Критическое чтение кода: понять, что делает блок, даже если не знаешь как его написать
- Умение формулировать ошибку: «тёмная тема не работает по кнопке» → точный баг
- Приоритизация: SEO раньше анимаций, архитектура раньше дизайна
