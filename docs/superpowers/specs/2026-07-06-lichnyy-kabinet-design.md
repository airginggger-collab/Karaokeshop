# Личный кабинет владельца — дизайн (вариант A: Git-CMS)

> Спек утверждён владельцем 2026-07-06: вариант **A (Git-CMS)** + вход через GitHub-аккаунт + начинаем с **Фазы 1** (вынос контента в файлы). Вариант A выбран как единственный, удовлетворяющий жёсткие рамки проекта (ADR-0002: ₸0, статика `output:export`, host-agnostic, без вендор-лока). Следующий шаг — план реализации Фазы 1 (writing-plans).

## Задача

Дать владелице (не-разработчик) редактировать весь контент сайта через безопасные формы в браузере, без правки «сырого» кода. Сейчас контент захардкожен в `apps/web/src/lib/site.ts` (TypeScript, ~880 строк) и правится напрямую на GitHub с риском «удалил кавычку — сломал сайт» ([docs/redaktirovanie-sajta.md](../redaktirovanie-sajta.md)). Личный кабинет = замена этого страшного редактирования на формы.

## Почему вариант A

Владелица **уже** правит контент через GitHub. Git-CMS — это апгрейд её нынешнего процесса: те же файлы в том же репозитории, но правки идут через формы (число — числовым полем, обязательные поля защищены), а не через ручную правку TS. Соблюдены все рамки ADR-0002: сайт остаётся чистой статикой (никакой миграции на SSR), бюджет ₸0 (CMS работает в браузере, бэкенд не нужен), контент остаётся в репозитории (нулевой вендор-лок), деплой — прежний (коммит в `main` → Cloudflare пересобирает ~1–2 мин). Отклонённые: **B** (свой Worker+БД — мгновенно и гибко, но большой объём, риск выйти из ₸0, SSR-миграция ломает ADR — избыточно для каталога дилера); **C** (Sanity/Storyblok — лучший редактор, но контент уходит к вендору + внешняя зависимость, против «host-agnostic / без лок-ина»).

## Архитектура

**CMS:** [Sveltia CMS](https://github.com/sveltia/sveltia-cms) — современный форк Decap CMS (Decap-совместимый `config.yml`, но легче, быстрее, лучше UX и проще с GitHub-аутентификацией). Один статический файл + конфиг. Fallback — Decap CMS, если со Sveltia возникнут проблемы (конфиг тот же).

**Поток данных:**
```
Владелица → /admin (статика, в браузере) → форма → коммит в GitHub (write через её GitHub-логин)
   → push в main → Cloudflare пересобирает сайт (~1–2 мин) → правка на проде
```

**Единый принцип:** контент — единственный источник в файлах-коллекциях; и сайт, и CMS читают ОДНИ и те же файлы. `site.ts` перестаёт быть хранилищем и становится тонким типизированным загрузчиком.

### Вынос контента из кода (ядро работы)

Контент из `site.ts` переезжает в файлы-коллекции под `apps/web/content/`:

| Коллекция | Файл(ы) | Тип в site.ts сейчас |
|---|---|---|
| Контакты/конфиг | `content/site-config.json` | `siteConfig` |
| Товары | `content/products.json` (или `content/products/*.json`) | `products: Product[]` (18) |
| Бренды | `content/brands.json` | `brands` |
| Блог | `content/blog/*.md` (frontmatter + тело) | `blogPosts: BlogPost[]` (26) |
| Истории установок | `content/stories.json` | `storyPosts` |
| Кейсы | `content/cases.json` | `cases` |
| Песни (образец) | `content/songs.json` | `songsSample` |
| Лендинг-мета/сценарии/комплекты | `content/landings.json` | `scenarios`, `bundles`, `staticPages`, `*Meta` |

`site.ts` → тонкий загрузчик: `import products from "../content/products.json"` и т.д., экспортирует те же имена и типы, что и сейчас. **Ни одна из ~70 страниц-потребителей не меняется** — контракт `site.ts` сохранён. Типы (`Product`, `BlogPost`, `Landing`, …) остаются в `site.ts`/отдельном `types.ts` и валидируют импорт. Блог как Markdown — тело статьи редактируется как текст, не как JS-строка.

### CMS-конфиг

`apps/web/public/admin/index.html` (грузит Sveltia CMS с CDN одним script-тегом) + `apps/web/public/admin/config.yml` описывает коллекции. `/admin` — часть статического билда, отдаётся Cloudflare как обычные ассеты (SPA грузится с CDN, конфиг читается рядом).

**Структура коллекций (решение 2026-07-07):** товары и блог — **folder-коллекции** (по файлу на запись: `content/products/<slug>.json`, `content/blog/<slug>.json`) → у владельца удобный список «выбери запись → правь форму», а не один гигантский список. Синглтоны (`site-config.json`, `page-meta.json`) — **file-коллекции** (один файл = набор полей). Остальные (stories, cases, songs, scenarios, bundles, brands, static-pages) — file-коллекции со списком (`list`-виджет), т.к. они мелкие. Это требует реструктуризации вывода Фазы 1: `products.json`/`blog.json` (массивы) → папки `content/products/`/`content/blog/` (по файлу на запись) + правка загрузчика `site.ts` (агрегировать файлы папки через `import.meta.glob`-аналог или сгенерированный индекс). Тест контент-целостности (29 тестов) стережёт, что после реструктуризации данные не потерялись.

Виджеты полей защищают от ошибок: `number` для цены, `boolean` для «в наличии»/флагов `SHOW_UNVERIFIED_*`, `image` для фото (медиатека), `select` для типа товара (`sistema/akustika/mikrofon/sub/miksher`), `required` для обязательных, `list` для `body`/`faq`/`kit`. Тело блога `body: string[]` → `list` из `text`-строк (абзацы), `faq` → `list` объектов `{q,a}`.

### Аутентификация — по личному токену (PAT), решение зафиксировано 2026-07-07

Sveltia CMS поддерживает вход по **personal access token** («Sign in with Token»): владелец жмёт кнопку на экране входа `/admin`, Sveltia ведёт на страницу генерации токена GitHub **с уже выбранными нужными правами**, владелец создаёт токен, вставляет — токен хранится в localStorage браузера. **Не нужны:** OAuth-приложение, OAuth-прокси Worker, правка config.yml (`base_url`). Это сохраняет всё в ₸0 и чистой статике (`/admin` = только `index.html` + `config.yml`, никакой второй серверной части). Подтверждено доками Sveltia (backend/github). Разовое действие владельца — сгенерировать токен (пошагово в `/admin-guide`).

OAuth (кнопка «Login with GitHub» без токена) — возможный апгрейд Фазы 3: добавляется деплоем `sveltia-cms-auth` Worker + `base_url` в config, БЕЗ переделки коллекций. Пока не делаем (YAGNI, лишняя инфраструктура). Простой пароль без GitHub в git-CMS невозможен (личность = GitHub) — это был бы откат к варианту B.

### Медиа (фото)

Медиатека Sveltia → загрузка коммитит файл в `apps/web/public/products/` (или `content/media/`), поле товара получает путь. Владелица грузит фото кнопкой, без ручной заливки в папку.

### Безопасность правок

- Типы полей CMS не дают внести «ломающую» правку (цена — только число, обязательные — не пустые).
- Каждая правка = обычный git-коммит → полная история и откат.
- **Контент-валидация на билде:** шаг проверки (all products: required-поля + `price` число + уникальные `slug`; JSON валиден) — битый контент роняет CI-билд, деплой не происходит, прод остаётся на прошлой рабочей версии. Добавляется в Phase 1.

## Фазы

**Фаза 1 — Контент в файлы (derisk, ценна сама по себе).** Вынести весь контент `site.ts` → типизированные JSON/MD; `site.ts` → тонкий загрузчик; поведение сайта идентично; 24 web-теста зелёные; +тест контент-целостности. CMS ещё нет — но владелица уже может править JSON через GitHub (безопаснее TS: нет логики, которую сломать). Отгружается отдельно.

**Фаза 2 — CMS-редактор.** (a) Реструктуризация `products.json`/`blog.json` (массивы) → folder-коллекции `content/products/`/`content/blog/` + правка загрузчика (тест целостности зелёный). (b) `/admin/index.html` + `config.yml` (все коллекции, PAT-вход). (c) Медиа: `apps/web/public/products/`. (d) Страница `/admin-guide/` из `docs/client/admin-guide.md`. (e) Разовое действие владельца: сгенерировать GitHub PAT и войти (проверка round-trip правки). Владелец редактирует формами.

**Фаза 3 — Полировка.** Брендинг `/admin`, preview-ссылки на страницы, валидация полей. Опционально: editorial-workflow (черновик → публикация).

**Страница-гайд `/admin-guide/` (deliverable Фазы 2 — вместе с рабочим `/admin`).** Владелец попросил (2026-07-06), чтобы инструкция «как заходить и работать в кабинете» открывалась прямо на сайте по адресу `/admin-guide/`. Текст гайда написан заранее — [`docs/client/admin-guide.md`](../../client/admin-guide.md) — и станет контентом страницы. Роут строится в Фазе 2 (не раньше: пока `/admin` не существует, гайд «зайдите в /admin» вёл бы на 404). Требования к странице: обычный роут `app/admin-guide/page.tsx` на компонентах «Сцены» (Breadcrumb, типографика), `<meta name="robots" content="noindex">` (служебная страница, не для поиска), контент из `admin-guide.md`. Заменяет `redaktirovanie-sajta.md` как основную инструкцию владельца.

Каждая фаза = свой цикл спек→план→реализация или отдельные задачи; Фаза 1 — приоритет и фундамент.

## Вне объёма

- Мгновенные обновления (осознанно ~1–2 мин, вариант A).
- Роли/много пользователей (один владелец).
- Лид-пайплайн (заявки в WhatsApp не сохраняются) — **отдельный трек**; git-CMS его НЕ решает (это про формы посетителей, не про контент). Остаётся в аудит-бэклоге.

## Критерии приёмки (Фаза 1)

- Весь контент вынесен из `site.ts` в `content/*`; `site.ts` не содержит данных, только загрузку+типы.
- `npm run build -w web` чист, 24 web + 2 ui теста зелёные, +тест контент-целостности зелёный.
- Прод визуально и по HTML идентичен до/после (чистый рефактор данных).
- CI/деплой зелёный.
